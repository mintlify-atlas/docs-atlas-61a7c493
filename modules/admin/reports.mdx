The Sistema de Admisión Web ships with a comprehensive reporting layer that covers every stage of the admissions lifecycle — from first pre-registration through final biometric check-in. Reports are available in two output formats: PDF (rendered by `barryvdh/laravel-dompdf` via the `Pdf` facade) and Excel (generated by `maatwebsite/excel`). All report endpoints are scoped to `auth()->user()->id_proceso`, so switching the active process automatically changes the data set without any additional filtering.

## Available Report Types

### 1. Applicant Roster by Program (Pre-inscriptions)

The Excel export controller (`ExcelController`) produces a grouped XLSX file containing one sheet-block per program. Each record includes DNI, full name, phone, email, program, modality, process, registration date, sex, inscription status, and any observations.

```bash
GET /admin/inscripciones/descargar-lista?programa=3&term=garcia
```

The file is streamed as `{date}preinscritos.xlsx`. When `auth()->user()->programas` is not null, the export is automatically scoped to the programs assigned to that admin user, enabling regional coordinators to download only their own programs.

### 2. Summary by Program (Pre-inscriptions vs Inscriptions)

`ReporteController::reportePrograma()` generates a side-by-side comparison of pre-inscription and inscription counts per program, including the numerical difference. Results can be returned as JSON for on-screen display or streamed directly as PDF.

```bash
# JSON response
POST /admin/reporte-programa

# PDF download
POST /admin/reporte-programa
Content-Type: application/json
{ "descargar": 1 }
```

**JSON response shape:**

```json
{
  "estado": true,
  "datos": [
    {
      "codigo": 3,
      "programa": "INGENIERÍA CIVIL",
      "inscripciones": 142,
      "preinscripciones": 189,
      "diferencia": 47
    }
  ],
  "totales": [
    {
      "total_inscripciones": 1840,
      "total_preinscripciones": 2310,
      "total_diferencia": 470
    }
  ]
}
```

PDFs are saved to `public/documentos/{id_proceso}/reportes/ReporteProgramas_{timestamp}{user_id}.pdf` before being streamed to the browser, creating a server-side audit copy.

### 3. Daily Inscription Report by Program

`ReporteController::reporteProgramaDiario()` pivots inscriptions by calendar date, generating one column per day that has at least one inscription. This is ideal for tracking registration momentum across the registration window.

```bash
# JSON
POST /admin/reporte-programa-diario

# PDF download
POST /admin/reporte-programa-diario
{ "descargar": 1 }
```

The view (`Reportes.programa_ins`) is rendered on A4 portrait orientation via `dompdf`.

### 4. Daily Inscription Report by User (Inscriptor Activity)

`ReporteController::reporteUsuarios()` pivots inscriptions by the staff member who registered them (`id_usuario`), producing the same daily-column structure. This allows management to audit individual inscriptor performance and identify any anomalies.

```bash
POST /admin/reporte-usuarios
{ "descargar": 1 }
```

### 5. Enrollment Summary (Resumen de Inscripciones)

`ResumenInscripcionesController::resumenInscripciones()` provides a flexible grouped summary of inscriptions. The caller specifies one or more grouping dimensions via the `group_by` parameter:

| Dimension key | Resolves to |
|---------------|-------------|
| `programa` | `programa.nombre` |
| `modalidad` | `modalidad.nombre` |
| `sexo` | `M` / `F` (derived from `postulante.sexo`) |
| `area` | `programa.area` |
| `usuario` | Full name of the staff user who registered the inscription |

```bash
# JSON — group by area and modality
POST /admin/resumen-inscripciones
Content-Type: application/json
{ "group_by": ["area", "modalidad"] }

# Excel download
POST /admin/resumen-inscripciones
{ "group_by": ["programa", "sexo"], "excel": 1 }

# PDF download
POST /admin/resumen-inscripciones
{ "group_by": ["programa", "modalidad"], "descargar": 1 }
```

The Excel file (`reporte.xlsx`) is generated by `ResumenInscripcionExport` using `maatwebsite/excel`. Column headings are automatically uppercased from the `group_by` keys.

PDF copies are written to `public/documentos/{id_proceso}/reportes-inscripcion/` before streaming.

### 6. Biometric Summary (Resumen Biométrico)

`ResumenBiometricoController::resumenBiometrico()` cross-references biometric captures in `control_biometrico` with puntajes/resultados to show how many admitted applicants have completed identity verification. It uses the same flexible `group_by` mechanism as the inscription summary.

```bash
# JSON — group by area and program
POST /admin/resumen-biometrico
Content-Type: application/json
{ "group_by": ["area", "programa"] }

# PDF download
POST /admin/resumen-biometrico
{ "group_by": ["modalidad", "sexo"], "descargar": 1 }
```

The response includes both the biometric counts per group and the `total_esperado` (approved applicants with `apto = 'SI'` from `puntajes`), broken down by area including a `TOTAL` rollup row using SQL's `WITH ROLLUP`.

PDFs are saved to `public/documentos/{id_proceso}/reportes-biometrico/`.

### 7. Demographic Reports (Dashboard Sub-panel)

Ten demographic breakdown endpoints are available under the admin prefix for on-screen chart data:

```bash
GET /admin/get-inscritos-genero-reporte          # Gender distribution
GET /admin/get-inscritos-edad-reporte             # Age distribution (top 7)
GET /admin/get-inscritos-residencia-reporte       # Residence by dep/prov/dist (top 6)
GET /admin/get-inscritos-procedencia-reporte      # Birth place (top 8)
GET /admin/get-inscritos-egreso-reporte           # Graduation year (top 7)
GET /admin/get-inscritos-discapacidad-reporte     # Disability flag
GET /admin/get-inscritos-tipo-documento-reporte   # Document type (DNI vs other)
GET /admin/get-inscritos-colegio-reporte          # Top 7 schools by cod_modular
GET /admin/get-inscritos-procedencia-colegio-reporte  # School location (top 7)
GET /admin/get-inscritos-tipo-colegio-reporte     # School management type
```

All ten endpoints return `{ "datos": [...], "estado": true }` and are consumed by the Chart.js visualisations in `Admin/Dashboard/components/reportes.vue`.

## Excel Export Details

The `maatwebsite/excel` package is used for all XLSX generation. Exports are invoked with:

```php
return Excel::download(new PreinscripcionExport($datos), date('d-m-Y H:i:s')."preinscritos.xlsx");
```

The `PreinscripcionExport` class receives a collection already grouped by program. Each group produces a labelled section in the spreadsheet. The `ResumenInscripcionExport` class dynamically maps the `group_by` headings to column headers.

## PDF Report Details

All PDF reports use `barryvdh/laravel-dompdf` with PHP execution and HTML5 parsing enabled:

```php
$pdf = Pdf::loadView('Reportes.programa_pre_ins', compact('res', 'proceso', 'totales'));
$pdf->getDomPDF()->set_option("isPhpEnabled", true);
$pdf->getDomPDF()->set_option("isHtml5ParserEnabled", true);
$pdf->setPaper('A4', 'portrait');
```

Blade templates for each report type live in `resources/views/Reportes/`:

| Template | Report |
|----------|--------|
| `programa_pre_ins.blade.php` | Pre-inscriptions vs inscriptions by program |
| `programa_ins.blade.php` | Daily inscriptions by program |
| `usuarios_inscripciones.blade.php` | Daily inscriptions by staff user |
| `inscripciones.blade.php` | Grouped inscription summary |
| `resumen_biometrico.blade.php` | Biometric summary |

Every PDF is written to disk before streaming, creating a persistent server-side archive at `public/documentos/{id_proceso}/reportes/`.

## Audit Trail Export

The audit trail can be browsed and filtered at `/admin/trazabilidad` and accessed programmatically:

```bash
# Paginated audit log (25 per page by default)
GET /admin/trazabilidad/data?action=UPDATE&date_from=2025-01-01&date_to=2025-01-31

# 30-day action summary
GET /admin/trazabilidad/resumen?id_proceso=14
```

The summary endpoint returns:
- `by_action` — count per action type (CREATE, UPDATE, DELETE, etc.)
- `by_model` — count per affected model (e.g. Vacante, Proceso, Documento)
- `top_actors` — top 10 users by action count with their display name or alias

<Tip>
  Enable audit trail review as part of your regular compliance workflow. Every INSERT, UPDATE, and DELETE performed by admin users is recorded in `audit_trail` with the user ID, alias, affected model, process ID, and a human-readable description. The 30-day resumen endpoint (`GET /admin/trazabilidad/resumen`) gives you a rapid summary of what changed and who changed it — essential for institutional accountability audits and for investigating any data discrepancies in published results.
</Tip>
